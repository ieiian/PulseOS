# PulseOS PRD v1

## 1. 产品概述

PulseOS 是一款移动健康类 APP，第一阶段聚焦五个核心模块：

- 用户
- 饮食
- 运动
- 冥想
- 睡眠

产品目标不是提供泛泛的健康资讯，而是帮助用户完成“今天该做什么”的健康行动闭环：

1. 输入基础信息
2. 设定健康目标
3. 记录日常行为
4. 获取规则判断与 AI 解释
5. 在首页看到今日进展与下一步建议

## 2. MVP 定义

### 2.1 第一版必须做

- 用户 onboarding
- 用户资料页
- 今日首页
- 饮食推荐与拍照分析
- 运动步数与心肺强化积分
- 冥想呼吸训练
- 睡眠录音与基础分析

### 2.2 第一版明确不做

- 社区
- 排行榜
- 医疗级报告
- 自建 AI 模型
- 复杂课程体系
- 硬件商城
- 多疾病深度诊断

## 3. 目标用户

### 用户画像 A：日常健康管理人群

- 年龄：22-40
- 特征：工作忙，作息不稳定，希望有简单明确的建议
- 需求：饮食提醒、运动目标、睡眠反馈

### 用户画像 B：减脂/体重管理用户

- 年龄：20-38
- 特征：关注热量、步数、作息与连续性
- 需求：饮食建议、快速记录、目标追踪

### 用户画像 C：轻度亚健康用户

- 年龄：25-45
- 特征：熬夜、久坐、偶尔焦虑
- 需求：睡眠分析、呼吸训练、温和提醒

## 4. 健康目标分类

- `fat_loss`：减脂
- `maintain`：维持健康
- `muscle_gain`：增肌
- `sleep_recovery`：作息恢复
- `stress_relief`：减压放松

说明：

- 第一版主目标仍以饮食、运动、睡眠三类为主
- `stress_relief` 主要影响冥想推荐与提醒文案

## 5. 核心价值主张

### 用户价值

- 减少记录成本
- 给出可执行建议
- 让健康数据更容易看懂
- 帮助建立持续行为习惯

### 产品价值

- 饮食模块承担 AI 差异化入口
- 运动模块承担日常高频使用
- 睡眠模块承担长期留存价值
- 用户模块承担长期配置与数据沉淀

## 6. 核心功能需求

### 6.1 用户模块

- 首次录入基础资料
- 编辑健康目标
- 编辑身体指标
- 查看个人统计数据
- 管理权限与通知
- 查看设备连接状态

### 6.2 饮食模块

- 当日饮食建议
- 断食计划展示
- 快速记录
- 常用餐复用
- 拍照分析
- 进食风险提示

### 6.3 运动模块

- 读取步数
- 计算心肺强化积分
- 展示日/周目标
- 手动补录
- 趋势展示

### 6.4 冥想模块

- 呼吸训练
- 基础音频播放
- 睡前快捷入口
- 时长统计

### 6.5 睡眠模块

- 睡眠开始/结束
- 夜间录音
- 呼噜/梦话事件标记
- 睡眠评分
- 睡眠建议

## 7. 关键非功能需求

- 优先保证可行性，不追求第一版极高算法精度
- Android/iOS 行为一致，业务规则以后端为准
- 页面结构清晰，重要操作 2 步内可达
- 配置、阈值、名称可统一管理
- 预留 AI provider、设备接入、算法升级空间

## 8. 可行性结论

### AI provider 结论

- 采用 API 接入是正确方案
- 不做自建模型
- 后端必须抽象 provider 层，避免绑定单一厂商

### 食物识别方案结论

- 第一版优先用成熟视觉识别 API
- 结果采用“识别结果 + 规则判断 + AI 解释”的三段式结构
- 不依赖自建食品库完成 MVP

### 睡眠方案结论

- 第一版只做基础录音与事件检测
- 不承诺医疗级准确度
- 睡眠分期、高精度识别后置

## 9. 成功标准

### 产品成功标准

- 用户能在首次使用 3 分钟内完成 onboarding
- 用户能在 10 秒内进入饮食拍照分析
- 用户能一眼看懂今天的运动完成度
- 用户能在醒来后快速看到昨夜睡眠总结

### 工程成功标准

- 后端规则可配置
- 双端目录边界清晰
- Phase 1 可以直接按计划进入脚手架搭建
