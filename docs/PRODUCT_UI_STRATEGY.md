# PulseOS 产品与 UI 策略

## 1. 目标

这份文档解决三个问题：

- 参考市面成熟产品后，PulseOS 的功能形态该如何取舍
- 界面风格该如何做到成熟、清晰、适合健康场景
- 后续扩展 AI、内容、设备接入时，结构如何不推倒重来

## 2. 市场借鉴结论

### 饮食

适合借鉴的方向：

- MyFitnessPal 的“快速记一餐”
- MyFitnessPal 的“常用餐 / 保存餐”
- MyFitnessPal 的“按时间记录”

不建议第一版就做的方向：

- 过度复杂的食物数据库运营
- 强依赖人工维护的大量品牌食品词条

### 运动

适合借鉴的方向：

- Apple Health 的摘要卡与趋势
- Google Fit 一类产品的周目标和积分式反馈

不建议第一版就做的方向：

- 大量社交挑战
- 跑步/骑行等高复杂 GPS 训练系统

### 冥想

适合借鉴的方向：

- Calm / Headspace 的低干扰导航
- 呼吸训练优先于复杂课程体系
- 睡前内容与夜间场景联动

不建议第一版就做的方向：

- 大规模内容订阅体系
- 多层级课程运营后台

### 睡眠

适合借鉴的方向：

- SnoreLab 的事件记录、趋势、重点片段回放
- Apple Health 的睡眠摘要展示

不建议第一版就做的方向：

- 声称医疗级分析
- 深度睡眠分期的强承诺

## 3. 产品定位优化

PulseOS 不应做成“功能很多但都很浅”的健康工具箱，而应做成：

- 一个以“今日行动”为核心的健康执行助手
- 一个以“记录 + 分析 + 提示”为闭环的个人健康面板
- 一个先重实用，再逐步增强 AI 的健康操作系统

这里建议明确一个产品原则：

- “我的”不只是容器页，而是完整的“用户模块”

原因：

- 基础资料、健康目标、统计数据、设置、权限、设备管理本身就是独立能力
- 后续接入会员、设备同步、数据导出时，用户模块会成为稳定扩展点
- 工程上也更容易形成 `user` 领域，而不是把各种杂项塞进 settings

推荐定位语气：

- 日常健康管理
- 个性化建议
- 科学但不过度医疗化
- 温和、可信、可执行

## 4. 功能架构优化

### 一级能力

- 今日首页
- 用户
- 饮食
- 运动
- 睡眠

### 二级能力

- 冥想
- 数据趋势
- 历史记录
- 设置与权限
- 健康档案
- 设备管理

### 核心闭环

1. 用户模块录入基础信息与目标
2. 系统生成目标与建议
3. 用户在健康模块中产生记录
4. 系统规则判断
5. AI 做解释与补充
6. 首页汇总并给出下一步动作
7. 用户模块沉淀统计、设置和长期管理

## 5. 首页设计策略

首页必须是“行动板”，不是“功能目录”。

首页建议按这个顺序排：

1. 今日概览
2. 今日总分
3. 当前最需要完成的一件事
4. 饮食 / 运动 / 睡眠摘要
5. 冥想快捷入口
6. 智能建议
7. 趋势摘要

这样做的好处：

- 用户一打开就知道该做什么
- 不会像很多健康 app 一样信息很多但缺乏主线
- AI 建议有明确承接位置

## 6. 模块页设计策略

### 饮食页

主目标：

- 尽可能降低记录成本
- 把“推荐”和“风险提示”放在最显眼的位置

推荐结构：

- 今日饮食目标卡
- 快速记录区
- 拍照识别入口
- 推荐餐单
- 常用餐
- 当日已记录时间轴

### 用户页

主目标：

- 统一承载身份、目标、设置、统计
- 让用户知道自己的健康状态和系统配置是否完整

推荐结构：

- 用户信息卡
- 健康目标卡
- 关键身体指标
- 周/月统计入口
- 设置与权限
- 设备与账号管理

### 运动页

主目标：

- 让用户清楚今天还差多少
- 用积分和周进度增强持续性

推荐结构：

- 今日步数与积分大卡
- 距离周目标差值
- 最近 7 天趋势
- 手动补录入口

### 冥想页

主目标：

- 低认知负担
- 快速开始

推荐结构：

- 一键开始呼吸训练
- 3 种基础呼吸模式
- 睡前冥想
- 最近使用

### 睡眠页

主目标：

- 睡前进入简单
- 醒来后结果清楚

推荐结构：

- 昨夜评分卡
- 睡眠时长与事件摘要
- 关键片段回放
- 趋势图
- 今晚建议

## 7. 视觉系统建议

### 品牌气质

- 可信
- 冷静
- 轻盈
- 不焦虑营销

用户模块视觉建议：

- 以中性专业风格为主，不要做成“账号中心”式工具页
- 用户统计卡的设计语言要和首页摘要一致
- 设置列表保持极简，避免和健康数据抢视觉注意力

### 颜色系统

- 主品牌色建议：蓝绿系中性色
- 模块辅助色分开
- 成功/提醒/风险使用语义色，不复用模块色

### 组件系统

- 卡片：统一 16-20 圆角
- 按钮：主按钮饱和度略高，次按钮轻描边
- 图标：线性或半填充风格统一
- 图表：少网格、少干扰、重结论

### 场景化风格

- 首页：中性专业
- 冥想：更柔和、更沉静
- 睡眠：更暗、更沉浸，但不强制全局深色

## 8. 动效建议

只保留有意义的动效：

- 首页卡片渐入
- 积分增长反馈
- 呼吸训练节奏动画
- 睡眠开始/结束过渡

避免：

- 大量漂浮装饰动画
- 复杂转场
- 高频反馈导致干扰

## 9. 模块化与扩展性建议

### 客户端

- 每个模块独立 feature 目录
- Design System 独立
- 网络、存储、权限单独沉淀在 core
- 不把后端规则复制到前端
- 用户模块独立 `user` feature，不和 settings/onboarding 混写

### 后端

- 模块按 domain 切分
- AI provider 抽象接口
- 规则引擎单独目录
- 评分系统统一抽象
- 定时任务与统计单独目录
- 用户资料、设置、设备、权限状态统一放 `user` 领域

### 产品扩展预留

- 穿戴设备接入
- 更多 AI provider
- 饮食数据库扩容
- 睡眠算法升级
- 会员体系或内容订阅
- 数据导出与隐私中心

## 10. MVP 优化建议

最优第一版：

- 首页总览
- onboarding
- 用户模块基础页
- 饮食推荐与拍照分析
- 运动步数与积分
- 冥想呼吸训练
- 睡眠录音与基础分析

最好不要在第一版加入：

- 社区
- 排行榜
- 医疗报告式输出
- 复杂商城或硬件绑定

## 11. 设计与研发协作建议

- 先出低保真信息架构，再出高保真
- 先定义 design token，再做页面
- 先定义 API 返回结构，再做双端页面绑定
- 每个模块都先做“空态、加载态、错误态”
- 用户模块优先定义资料结构与设置结构，否则后面模块会反复返工

## 12. 最终建议

PulseOS 第一阶段最重要的不是“功能最多”，而是：

- 首页有没有主线
- 饮食记录够不够顺手
- 运动积分有没有持续反馈
- 睡眠结果是不是一眼能懂
- UI 是否达到成熟产品的秩序感

只要这五点成立，后面的 AI、内容、算法增强都会顺很多。
